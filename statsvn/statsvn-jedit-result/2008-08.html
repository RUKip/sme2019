SF Bug #2053299: ProjectViewer 2.9: Confused about symlinks

It's not really PV who is confused, but combination of VFS reading the link before the real path, and the symlink loop detection code being way too strict. The loop detection logic should only be applied when resolving symlinks; symlinks that point to other directories in the same tree but don't form a loop can be safely traversed.

This causes the traversal to return more data than before in certain cases, but it's returning the correct data given the user's filesystem.

Removed the association between a view's persistence in perspective.xml and its layout file. The layout file is produced and read by the docking framework for each view according to the view index, removing the need for an association in the perspective file. This way, when the docking framework is switched, we can use the previous layout file that was created for the view by the framework to which we switched.
Also, the saving and loading of named layouts no longer requires a perspective file in a separate directory; only the framework directory is needed. The list of named layouts to create dynamic actions for is now taken from the framework, ensure that we don't create dynamic actions for layouts saved in other frameworks.

Also, added some handling of plugin update messages on the edit bus, though this is highly experimental at this stage, sort of a copy of the former implementation.

shlomy 2008-08-10 21:54 #29 Rev.: 13259

Merging my docking_framework branch back into the main trunk. This adds a few new features to the jEdit core:
1. A pluggable docking framework, which can be provided as a service by plugins. To choose the docking framework, use Global Options -> Docking. The original jEdit dockable window manager is named 'Original'. If you get the MyDoggyPlugin plugin, you can select between Original and MyDoggy.
2. Named perspectives - you can now save the docking layout in a file, and load it any time later from a file. Also, once you save the layout in a file, you get an action that loads the layout from this file, and you can bind this action to a keyboard shortcut or toolbar button just like any other action. The actions to load/save the layout are available through the View->Docking menu, and the action set for keyboard shortcuts or toolbar buttons is named "Docking layouts" (in Global Options dialog).
3. Mode-specific perspectives - you can associate the docking layout with an edit mode, each edit mode having its own layout. You can have jEdit automatically save the layout of an edit mode when you switch to another, and automatically load the layout of an edit mode when you enter the mode. The auto load/save of mode-specific layout can be set using Global Options -> Docking.

Implementation-wise, several major changes were done to support a pluggable docking framework:
1. Management of toolbars and split configuration (of edit panes) is managed entirely by the core - the docking framework is unaware of it. The framework just gets a panel to be used as the center component, and the core sets its contents with the split configuration and the toolbars when needed.
2. The docking layout has been abstracted out of ViewConfig, and is now delegated to the docking framework for persistence.
3. A new DockingLayoutManager class listens to edit bus messages and responds to edit mode changes by saving/loading the docking layout if the appropriate options are set.
